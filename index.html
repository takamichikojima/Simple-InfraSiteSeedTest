※ 上記基準の根拠

  • Download — 総務省「令和5年 通信利用動向調査」によれば固定ブロードバンドの平均実測速度は約 100〜300 Mbps であり、社内 LAN (1 GbE) では 100 Mbps 以上が快適な基準となります。IEEE 802.3ab (1000BASE-T) の理論値 1 Gbps に対し、実効スループットは 30〜70% 程度が一般的です。
  • Latency (RTT) — Cisco の QoS 設計ガイドでは、インタラクティブ通信の推奨 RTT は 150 ms 以下、社内 LAN では通常 1〜5 ms 程度です。ITU-T G.114 勧告では片道 150 ms (往復 300 ms) を音声品質の上限としています。10 ms 以下を「優秀」としたのは、同一セグメント内 LAN の典型値に基づきます。
  • Jitter — Cisco および ITU-T Y.1541 では、VoIP の許容ジッターは 30 ms 以下を推奨しています。社内ネットワークでは 5 ms 以下が安定とみなされ、ビデオ会議等のリアルタイム通信でも問題が生じない水準です。

出典: ITU-T G.114 / Y.1541, Cisco QoS Design Guide, IEEE 802.3ab, 総務省 「令和5年 通信利用動向調査」

【追加調査】tracert による通信経路トレース

1. tracert とは

クライアント PC からサーバーまでの通信経路上にある各ルーター（ホップ）を特定し、各区間の応答時間を表示するコマンドです。ネットワーク遅延のボトルネック箇所を切り分ける際に使用します。

2. 実行方法

Windows の コマンドプロンプト または PowerShell を開き、以下を実行してください。

tracert <サーバーのホスト名 or IPアドレス>

例:
tracert 192.168.1.100
tracert aras-server.local

3. 出力の読み方

  1     1 ms    <1 ms    <1 ms   192.168.1.1      ← デフォルトゲートウェイ (自席のルーター)
  2     2 ms     1 ms     2 ms   10.0.0.1         ← コアスイッチ / L3スイッチ
  3     3 ms     3 ms     2 ms   10.0.1.1         ← サーバーセグメントのルーター
  4     4 ms     3 ms     3 ms   192.168.1.100    ← 宛先サーバー
  1. 左端の番号 — ホップ番号（経由するルーターの順番）
  2. 3つの時間 (ms) — そのホップまでの往復時間を3回計測した値。数値が大きいほど遅延が大きい
  3. 右端の IP / ホスト名 — 経由しているネットワーク機器のアドレス

4. 結果の判断基準

  • 特定ホップだけ急に遅延が増加 → その区間のネットワーク機器・回線に問題がある可能性
  • * * * (タイムアウト) が表示される → その機器が ICMP 応答を無効化している場合が多く、必ずしも障害ではない
  • 最終ホップに到達しない → ファイアウォール等でブロックされている、またはサーバーがダウンしている可能性
  • 社内 LAN で全ホップ 5 ms 以下 → 正常な範囲

※ 上記基準の根拠

  • ホップ間遅延の急増 — RFC 2151「A Primer On Internet and TCP/IP Tools and Utilities」では、traceroute の各ホップ間 RTT の急増を「ボトルネック候補」として注目すべきと記述されています。Cisco のトラブルシューティングガイドでも、前ホップとの差分が 30 ms 以上ある場合にそのセグメントを重点調査対象としています。
  • * * * (タイムアウト) — RFC 1812「Requirements for IP Version 4 Routers」§4.3.2.8 では、ルーターが ICMP Time Exceeded メッセージの送信をレート制限または無効化できると規定しており、タイムアウトが即障害を意味しない根拠となっています。多くの企業ネットワーク機器（Cisco, Juniper 等）はセキュリティポリシーにより ICMP 応答を抑制しています。
  • 最終ホップ未到達 — Microsoft TechNet「Understanding Tracert」および Cisco「Using Traceroute to Troubleshoot」では、最終宛先に到達しないケースとして、ファイアウォールの ICMP フィルタリング、ACL によるブロック、またはホストダウンを主要原因として挙げています。
  • 社内 LAN 5 ms 以下 — IEEE 802.3ab (1000BASE-T) の同一セグメント内スイッチング遅延は通常 10〜100 μs であり、複数ホップを経由しても社内 LAN 内では 1〜5 ms が標準的です。Cisco の Campus Network Design Guide では、アクセス層からコア層までの End-to-End 遅延が 5 ms 未満であることを推奨しています。ITU-T G.114 勧告の片道 150 ms 上限と比較しても十分に余裕がある水準です。

出典: RFC 1812, RFC 2151, Cisco Campus Network Design Guide, Cisco Troubleshooting Guide, IEEE 802.3ab, ITU-T G.114, Microsoft TechNet

5. 追加オプション

tracert -d 192.168.1.100      DNSの逆引きを省略（高速化）
tracert -h 15 192.168.1.100   最大ホップ数を15に制限（既定は30）
tracert -w 1000 192.168.1.100 各ホップのタイムアウトを1000msに設定
ヒント: 本ページの計測結果と合わせて tracert の結果をネットワーク管理者へ共有すると、遅延箇所の特定がスムーズになります。結果はコマンドプロンプトで右クリック → 「すべて選択」→ Enter でコピーできます。
